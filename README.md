# 📄 Правила ведения репозитория

## 1. Ветви

### Основные ветви

* `main` – ветвь, содержащая конечные версии проекта.
* `develop` – ветвь, содержащая текущую разарабатываемую версию проекта.

### Дополнительные ветви

Дополнительные ветви оформляются в виде `type/#number`:
* `type` – вид задания:
  * `task` – задание на разработку с доски.
  * `bugfix` – задание на исправление бага с доски.
* `#number` – номер задания с доски.

> Примеры: `task/#42`, `bugfix/#67`.

## 2. Issues

При выполнении задания по необходимости создается `issue` для обсуждения. При создании необходимо:
* Оформить название в соответствии формату `[<название_ветви>] <тема_обсуждения>`.
* Назначить себя в поле `assignee`.
* Установить тип обсуждаемого задания в `label`.

> Примеры: `[task/#42] Реализация интерфейса IUserService`.

## 3. Коммиты 

Коммиты производятся после каждого законченного семантически насыщенного действия. Названия коммитов соответствуют формату `(<вид_изменения>) <описание>`:
* Виды изменения:
  * `+` – добавление/исправление.
  * `-` – удаление.
* Описание коммита должно быть исчерпывающим, но неизбыточным.

> Примеры: `(+) Реализовал интерфейс IUserService`, `(+) IUserService`, `(-) Удалил интерфейс IUserService`, `(-) IUserService`.

## 4. Запросы на слияние 

Все запросы на слияние производятся в основную ветку `develop`. Для проведения слияния веток необходимо:
* Оформить название запроса на слияние в соответствии формату `[<название_ветви>] <описание_изменений>`.
* Назначить требуемых людей в поле `reviewers`.
* Назначить себя в поле `assignee`.
* Установить тип выполненного задания в `label`.
* Обеспечить наличие положительно пройденного автоматического тестирования.
* Обеспечить отсутствие ошибок слияния веток.
* Обеспечить наличие одобрения минимум **одного** члена команды.

При отклонении запроса его необходимо пометить как **черновик**.

> Примеры: `[task/#42] Реализация функционала регистрации/авторизации`, `[bugfix/#67] Исправление ошибки в работе функционала регистрации/авторизации`.
